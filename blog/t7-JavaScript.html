Javascript versus Ruby

February 7 2014

This week was my cohort's first chance to use Javascript. My first impression of Javascript was that it seemed nimble, by that I mean that the syntax by comparison to Ruby was less rigid. Ruby has a lot of explicit declarations for things such as classes and definition of methods. On top of this, there are explicit order of operations when using built in methods such as ".each" on arrays and enumerables. Javascript on the other hand seemed to be very permissive changing how and when to use functions (the equivelant of defining methods for Ruby). What does this mean? It means that functions seem to be the universal form of declaring a set of actions with no real order of operations like that of ".each" or formal declarations other then the calling of the function itself.

I'm going to try to use an example in both Ruby and Javascript on this one. This week we had an assignment to create a game on Javascript. I wanted to keep the game simple but applicable to a game in the real world. I chose to make a blackjack game. The first example I'll show you is how I created a blackjack game on Ruby then I'll compare that to my blackjackgame in Javascript.

In Ruby, the first thing I did was create a class to indicate that the object that I was creating was a "Game" class. I then stated that within that class I want 2 actions / methods to be called upon. I want the class to "play" to symbolize the action a player takes while playing Blackjack. For example the player will either "hit" or "hold". The other action is for the class to "deal". This symbolizes the actions of the players opponent "the dealer" to deal his hand for later comparison against "the player."

I then created a second class called "Winner" to determine the winner between the "player" and the "dealer". I used composition style development to initialize a new "Game" class object within my "Winner" class. This is in order to allow me access to "Game" class methods from with the "Winner" class.

I then created a method within the "Winner" class which compared the values returned from the method "game.play" representing "the players hand" and the method "game.deal" representing "the dealers hand". I then stored those values in instance variables and ran them through conditional statements to determine my winner.

The reason I believe JS seems lighter is because my variables and functions can be called upon with needing to create a new object within my function to give me access to those variables and functions. This may seem confusing but bare with me on this one.

What I'm refering to is the un-necessity for me to create a new object that declares its class before I am able to access its methods. This is referring to my composition design. My "play()" function and "deal()" function were ran in my "winner()" function without needing to say "randomObject = thisClass.new". This also meant that I didn't have to create new variables to set equal to the other classes methods in order to return their values ie: "@phand = @game.play" or "@dhand = @game.deal". I merely had to call upon the variable that stored those values in this case "playerHand" or "dealerHand".

I think the agile coding in Javascript is nice for simplicity and ease of development however I do so see the virtues of creating explicit classes and methods that are only accessible within those confines. I would go on to say that Javascript while being more agile would require the programmer to mindful to foresee possible conflicting messages in their programs that Ruby developers may not have to think about due to the design of the language however in terms of agility and ease of use I would say Javascript currently holds that title.
